# PSS Prestige Optimizer

一个纯前端的 Pixel Starships Prestige / 船员合成路线优化器。用户输入当前仓库库存和七星目标后，程序会在浏览器本地计算近似最优合成路线，并展示路线阶段、资源消耗、瓶颈、缓存命中、剪枝状态和搜索耗时。

项目当前是单文件应用：

```text
最优化合成路线计算器- w1n9.html
```

不需要后端、不需要安装依赖、不需要构建步骤。

## 使用方法

### 1. 打开页面

直接用浏览器打开 HTML 文件：

```text
最优化合成路线计算器- w1n9.html
```

推荐使用 Chrome、Edge、Safari 等现代浏览器。页面会在浏览器本地完成所有计算。

### 2. 录入仓库库存

在“仓库”区域添加你当前拥有的 1-5 星船员数量。

注意：

- 规划器只把 1-5 星船员作为可消耗库存。
- 六星船员不参与常规合成规划。
- 七星船员只作为目标，不继续作为原料。
- 输入数量会自动保存到浏览器本地。

### 3. 添加七星目标

在“目标”区域选择要合成的七星船员，然后设置：

- `优先级`：数字越小优先级越高。
- `期望数量`：希望至少合成多少个；留空表示不限。

点击“添加目标”后，目标会进入规划清单。可以添加多个七星目标一起规划。

### 4. 选择优化模式

当前支持两种模式：

- `期望优先`：优先满足高优先级目标的期望数量，然后再考虑更多产量。
- `最大数量`：优先让七星总产量最大化。

如果你明确知道最想要哪些目标，建议使用“期望优先”。如果只想最大化可合成的七星数量，使用“最大数量”。

### 5. 设置搜索宽度

搜索宽度控制搜索保留的候选状态数量：

- `60`：速度快，适合目标少或快速试算。
- `120`：默认平衡档。
- `300`：更慢，但通常更接近全局最优。

搜索目标多、库存复杂或宽度高时，可能需要几十秒。搜索阶段页面会显示“搜索中”提示，并临时禁用生成按钮。

### 6. 生成方案

点击“生成近似最优方案”后，结果区域会显示：

- 近似产量。
- 期望满足情况。
- 搜索耗时和是否截断。
- 每个目标的单目标路线池规模。
- 最终选中的路线。
- 主要消耗。
- 关键瓶颈。
- 按星级分组的合成步骤。

结果是近似最优，不保证严格数学全局最优。

### 7. 保存计划和冲突分析

生成方案后可以保存计划。保存后的计划可以用于分析多个计划之间是否会争抢同一批船员。

冲突分析会显示：

- 哪些船员库存不够。
- 每个冲突资源被哪些计划消耗。
- 当前库存执行多个计划后的剩余情况。

### 8. 导入和导出

页面支持导出 JSON 存档，建议在清理浏览器缓存、换设备或大规模修改库存前备份。

导出的数据包括：

- 当前仓库库存。
- 已保存计划。
- 当前七星目标清单。
- 优化模式。
- 搜索宽度。

## 数据保存

项目没有后端服务，也不会上传数据。所有数据都保存在当前浏览器本地。

使用的本地存储：

- `localStorage`：主存储，key 为 `fleet-warehouse-manager-v1`。
- `IndexedDB`：辅助持久化存储，数据库名为 `fleet-warehouse-manager`，对象仓库为 `state`。

自动保存内容包括：

- `warehouse`：仓库库存。
- `savedPlans`：已保存计划。
- `optimizationTargets`：七星目标清单。
- `optimizationMode`：优化模式。
- `optimizerBeamWidth`：搜索宽度。
- `currentOptimization`：最近一次计算结果。
- 筛选条件和辅助查询状态。

如果更换浏览器、清理站点数据或切换设备，本地数据不会自动同步。

## 算法原理

### 问题建模

每条合成配方可以理解为：

```text
原料 A + 原料 B -> 产物 C
```

用户给定：

- 当前拥有的船员库存。
- 一个或多个七星目标。
- 每个目标的优先级和期望数量。

程序要找一组配方执行次数，使得：

- 总消耗不超过库存。
- 产出的七星目标尽量满足用户目标。
- 在相同产量下尽量减少稀缺资源消耗。

由于配方数量和组合分支很多，完整枚举所有路线会造成状态爆炸。因此当前实现采用近似优化，而不是严格全局最优求解。

### 配方图

程序启动时会把静态配方数据转换为 `RECIPE_BOOK`：

- `byKey`：通过 `原料A|原料B->产物` 定位唯一配方。
- `byOutput`：通过产物 ID 找到所有可以合成它的配方。
- `craftableIds`：所有可合成船员 ID。

两个原料会排序后再生成 key，所以：

```text
A + B -> C
B + A -> C
```

会被视为同一条配方。

### 闭包

这里的“闭包”是配方图上的可达子图，不是 JavaScript 函数闭包。

从某个七星目标出发，递归查找：

```text
谁能合成这个目标
这些原料如果也能被合成，又分别由谁合成
继续往下直到不能再展开
```

最终得到的相关船员和配方集合，就是该目标的配方闭包。

闭包的作用是缩小搜索范围。规划某个七星目标时，不需要扫描全量配方，只搜索和该目标有关的子图。

### 单目标路线池

对每个七星目标，程序会先构建一个单目标路线池。

路线池记录：

```text
目标 X 合成 1 个的若干候选路线
目标 X 合成 2 个的若干候选路线
目标 X 合成 3 个的若干候选路线
...
```

每条路线包含：

- 需要执行的配方次数。
- 从仓库净消耗的船员。
- 执行后的剩余库存。
- 路线成本。
- 瓶颈资源。

路线池通过 beam search 生成。每一轮尝试在当前状态上再合成一个目标，只保留成本较低、消耗更优的一部分候选状态。

### 递归提供原料

合成一个目标需要先“提供”它的两个原料。程序使用递归函数寻找某个船员：

```text
如果仓库里有该船员：
  直接消耗 1 个
否则如果该船员可以合成：
  尝试它的候选配方
  递归提供配方的两个原料
否则：
  该路线不可行
```

递归会受到深度、候选配方数量、分支数量和时间预算限制，避免搜索无限扩大。

### 候选配方评分

配方候选不会全部展开，而是先打分排序。当前评分综合考虑：

- 当前库存是否直接满足原料。
- 原料库存数量。
- 原料是否稀缺。
- 原料继续合成的估计深度。
- 原料星级。

直觉上，算法会优先选择：

- 当前库存已经能直接做的配方。
- 不过度消耗稀缺原料的配方。
- 合成链较短的配方。
- 成本更低、风险更小的路线。

### 上界剪枝

在单目标搜索阶段，程序会估计当前剩余库存理论上还能否继续合成该目标。

如果估算结果显示某个状态已经不可能再产出目标，就提前跳过该状态。

这能减少明显无效的搜索分支，尤其在库存已被消耗到瓶颈资源不足时更有效。

### 多目标组合优化

单目标路线池生成后，程序会在多个目标之间做组合优化。

例如：

```text
目标 A 可选路线 A0, A1, A2
目标 B 可选路线 B0, B1, B2
目标 C 可选路线 C0, C1, C2
```

组合优化会从每个目标中选择一条路线，要求总消耗不超过当前仓库库存。

这个阶段也使用 beam search：

- 逐个目标扩展候选状态。
- 丢弃资源消耗超过剩余库存的组合。
- 对剩余组合计算评分。
- 去重并保留前 `beamWidth` 个状态。

### 评分函数

`期望优先` 模式下，评分重点是：

- 高优先级目标是否满足期望数量。
- 未满足期望会扣分。
- 超出期望也加分，但权重较低。
- 最后扣除一部分资源消耗成本。

`最大数量` 模式下，评分重点是：

- 七星目标总产量。
- 同产量下再比较优先级和资源成本。

资源成本会考虑：

- 消耗数量。
- 船员星级。
- 当前拥有数量。

库存越少、星级越高的船员，消耗成本越高。

### 缓存设计

当前有两类算法缓存：

- 目标闭包缓存：相同目标集合不重复构建闭包。
- 路线池缓存：相同目标、相同库存签名、相同搜索宽度时复用路线池。

路线池缓存是内存缓存，刷新页面后会丢失。它主要用于同一页面会话中反复调整优先级、期望数量或重新计算的场景。

## 程序设计

### 技术架构

项目是单文件前端应用，主要由三层组成：

```text
HTML/CSS
  页面结构和视觉样式

静态数据
  船员、星级、配方、拼音首字母

JavaScript
  数据清洗、算法搜索、状态持久化、页面渲染和事件绑定
```

不依赖第三方库。

### 核心模块

虽然当前代码在一个 HTML 文件内，但逻辑上可以分为：

#### 1. crew-data

保存静态数据：

- 船员 ID。
- 船员名称。
- 船员星级。
- 星级分组。
- 合成配方。
- 拼音首字母索引。

#### 2. planner-core

负责算法和数据计算：

- 构建配方图。
- 构建目标闭包。
- 递归生成候选路线。
- 生成单目标路线池。
- 多目标路线组合优化。
- 计算消耗、剩余、产出和瓶颈。
- 维护搜索缓存和搜索统计。

#### 3. app

负责页面状态和交互：

- 仓库输入。
- 目标管理。
- 规划触发。
- 结果渲染。
- 计划保存。
- 冲突分析。
- 辅助配方查询。
- 导入导出。
- localStorage / IndexedDB 持久化。

### 状态管理

核心状态集中在 `state` 对象中，包括：

- 仓库库存。
- 已保存计划。
- 当前目标列表。
- 当前优化结果。
- 搜索模式和搜索宽度。
- UI 筛选条件。
- 辅助查询状态。

状态变化后会调用 `persistState()` 自动保存，并触发必要的重新渲染。

### 渲染方式

当前使用原生 DOM API 和模板字符串渲染，不使用 React/Vue。

主要渲染函数包括：

- `renderWarehouse()`：仓库选择和库存展示。
- `renderOptimizationTargets()`：七星目标清单。
- `renderPlannerResult()`：规划结果、搜索状态和合成步骤。
- `renderSavedPlans()`：已保存计划。
- `renderAnalysis()`：计划冲突分析。
- `renderHelperResults()`：辅助配方查询结果。

### UI/UX 设计

页面采用统一的浅色卡片式界面：

- 统一使用 `panel`、`summary-card`、`metric-card`、`info-chip` 等组件风格。
- 搜索阶段会显示明确的 busy 状态和旋转提示。
- 长时间搜索时会禁用相关按钮，避免重复触发。
- 搜索结果会显示是否截断、缓存命中和剪枝数量，降低用户等待时的不确定感。

## 限制

当前实现是浏览器内的近似优化器，主要限制包括：

- 不保证严格数学全局最优。
- 搜索宽度越大越慢。
- 路线池缓存只在当前页面会话中有效。
